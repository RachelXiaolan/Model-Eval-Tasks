## ⚠️ 公平性声明（请先读）

本次测试存在**两个影响公平性的关键问题**，结论需谨慎解读：

### 1. D7-B（抗幻觉测试）并非 Fable 5 原生回答
Fable 5 在回答 D7-B 时触发了平台安全机制（`Fable 5's safeguards flagged this message`），**Claude Desktop 自动切换到 Opus 4.8 代为回答**。因此：
- D7-B 的优秀表现（识破了 2 个陷阱题）**应归功于 Opus 4.8，不计入 Fable 5 成绩**。
- Fable 5 在 D7-B 上的真实表现 = **触发 safeguard，拒绝/中断回答**。这本身是一个值得记录的可靠性观察点。
- 本报告中 D7 维度得分**仅基于 D7-A**，D7-B 标记为「不可比」。

### 2. GLM-5.2 因环境/协议问题未能完成评测
GLM-5.2 通过 Codespace + Claude Code Router 接入时，复杂请求会触发 `Request timed out`。根因是 GLM 的 `reasoning_content` 流式输出与 Router 协议转换层不兼容。**GLM-5.2 本次无法产出可评测的有效结果**，详见第 4 节。

> **结论**：本报告的评分**仅适用于 Fable 5**（D1/D2/D7-A）。GLM-5.2 需更换接入方式后重测。Fable 5 与 GLM-5.2 之间**暂无可比分**。

## 四、GLM-5.2 测试失败记录

### 4.1 环境配置

测试在 GitHub Codespace（`glm-claude-code`）中进行：
- Claude Code CLI v2.1.201
- 引入 **Claude Code Router** 作为协议转换层（Anthropic Messages API ↔ OpenAI 兼容 `/chat/completions`）
- GLM-5.2 接口：`https://open.bigmodel.cn/api/paas/v4/chat/completions`

### 4.2 失败现象

| 场景 | 结果 |
|------|------|
| 简单 prompt（"只回复 OK"） | ✅ 正常返回 |
| 复杂 prompt（D1 愤怒的小鸟等） | ❌ `Request timed out` |
| 官网体验中心（网页端） | ⚠️ 能返回代码，但无法保存文件，刷新后历史丢失 |

### 4.3 根因分析（Codex 排查结论）

复杂请求时，GLM-5.2 会**持续输出 `reasoning_content`**（思维链）。Claude Code Router 对这种 reasoning 流的处理不稳定，导致 Claude Code 端长时间等待并最终超时。

**临时绕过方案**（Codex 尝试）：通过 Router transformer 关闭 GLM 的 thinking 并限制输出：
```
["reasoning", { "enable": false }]
["sampling", { "max_tokens": 4096, "temperature": 0.7 }]
```
但关闭 thinking 后已无法公平评测 GLM-5.2 的完整能力（GLM-5.2 的 reasoning 是其核心特性）。

### 4.4 费用消耗去向分析

> 以下是测试人对 GLM 账单明细的核查结论（基于智谱开放平台费用导出）。

测试人充值 ¥20，本次测试**净消耗约 ¥21**（含信用额度补足）。去向分布：

| 消费来源 | 金额 | 占比 | 说明 |
|---------|------|------|------|
| **Codespace 卡死空转 + 官网体验中心** | ≈¥19.7 | ~93% | 18 笔「大额输出」订单（单笔 2–8 万 token），是 GLM 在复杂请求下持续输出 reasoning、客户端已超时但 token 持续燃烧所致 |
| Codex 连通性测试（4 笔简单测试） | ≈¥0.33 | ~1.5% | 与 Codex 自述完全吻合 |
| Codex 另一轮排查测试（含联网搜索） | ≈¥1.15 | ~5.5% | 排查 reasoning 流问题时产生 |

**核心结论**：Codex 声称的「只消耗 ¥0.33」属实——那 4 笔简单测试确实只花了这么多。**¥20 的大头（约 93%）烧在「GLM reasoning 流式输出 + Router 不兼容导致的客户端超时空转」**，即模型端一直在生成 token，而 Claude Code 端以为请求失败了。这是 GLM-5.2 经第三方 Router 接入时的一个**实际工程陷阱**。

> ⚠️ 这不是 GLM-5.2 模型能力的问题，而是**接入层协议兼容性**问题。GLM-5.2 的真实能力需用原生网页端或官方 SDK 重测。

### 4.5 GLM-5.2 重测建议

| 方案 | 可行性 | 说明 |
|------|--------|------|
| ① GLM 官方网页端逐题测 | ⭐推荐 | 最简单，但无法保存文件（需手动复制），且长输出易丢失 |
| ② 智谱官方 SDK / zhipuai 库 | ⭐推荐 | 脚本化、可控、能保存输出，绕过 Router |
| ③ 调大 Router 超时 + 保留 thinking | 可尝试 | 治标不治本，reasoning 流稳定性仍存疑 |
| ④ 关闭 thinking 测 | ❌ 不推荐 | 无法公平评测 GLM-5.2（reasoning 是其卖点） |

## 五、阶段性结论

### Fable 5（已测 3 维度）
- **推理与知识极强**：D2 满分（会主动验算、指出题目漏洞），D7-A 满分（全对 + 识破皮肤陷阱）。
- **代码生成扎实**：D1 功能完整能跑，但界面偏朴素、生成过程偶有 `request failed`。
- **平台局限**：D7-B 被 Claude Desktop 的 safeguard 切换到 Opus 4.8，说明在平台侧测 Fable 5 时，某些 prompt 会被安全策略拦截，**建议后续用 API 直连重测 D7-B**。

### GLM-5.2
- 本次**因接入层问题无法评测**，不产生任何能力分。
- 暴露的工程问题（reasoning 流 + Router 不兼容 → 超时空转烧钱）对实际使用者有参考价值。
- 建议用官方 SDK 重测。
