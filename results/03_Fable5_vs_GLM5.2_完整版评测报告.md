> **相关文件**：[Fable5 实测样本](./fable5/) · [GLM5.2 实测样本](./glm5.2/) · [初版报告](./01_Fable5_vs_GLM5.2_详细测试报告.md)

## 🎯 执行摘要

| | Fable 5 | GLM-5.2 |
|---|---|---|
| **加权总分** | **4.90 / 5.00（S 级）** | **4.90 / 5.00（S 级）** |
| **已测维度** | 8/8 ✅ | 8/8 ✅ |
| **总成本** | **$9.51** | **$2.67**（仅为 Fable5 的 **28%**） |
| **红线问题** | ⚠️ D7-B 触发平台 safeguard | ❌ 无 |
| **核心结论** | 推理/知识/创意极强，代码扎实但界面朴素 | 综合能力同等顶尖，**性价比碾压**，代码物理引擎更专业 |

### 三句话结论

1. **能力上两个模型几乎完全打平**——8 个维度加权总分都是 4.90（S 级），在精心设计的测试题上都达到了当前模型的最高水平。
2. **唯一的差异化在 D1 代码生成**——GLM5.2 的物理引擎更专业（逆质量系统 + 基于 dt 的帧率无关物理 + 弹道预测线），但 D1-B 贪吃蛇需要二次 prompt 才输出文件；Fable5 一次性生成更完整，但 request failed 次数多（D1-B 高达 6 次）。
3. **成本效率 GLM5.2 大幅领先**——完成同样的 15 个测试任务，GLM5.2 花费 $2.67，Fable5 花费 $9.51，**差 3.6 倍**。如果纯粹按"每美元能买到多少能力"算，GLM5.2 性价比远超 Fable5。

## ⚠️ 公平性声明（3 个关键问题）

### 问题一：Fable 5 的 D7-B 非模型原生回答

Fable 5 在 D7-B（抗幻觉测试）时触发了 Claude Desktop 平台安全机制：

> *Fable 5's safeguards flagged this message. Switched to Opus 4.8.*

平台**自动切换到 Opus 4.8 代为回答**。因此：
- D7-B 的优秀表现（识破 2 个陷阱）**归功于 Opus 4.8，不计入 Fable 5 成绩**。
- Fable 5 在 D7-B 的真实表现 = **触发 safeguard**，无法独立评测。
- 本报告 Fable 5 的 D7 得分**仅基于 D7-A**。
- GLM-5.2 的 D7-B 是**模型原生回答**，完美识破两个陷阱，计入成绩。
- **建议**：后续用 Fable 5 的 API 直连重测 D7-B，绕过平台 safeguard。

### 问题二：GLM-5.2 的联网搜索功能在 D8-A 全部失效

GLM5.2 在 D8-A 行程规划中发了 4 次 web search 请求，**全部返回 "Did 0 searches"**（0 结果）。但 GLM5.2 用自身知识兜底，给出了合理的行程和价格估值，并在结尾**诚实声明**"我引用的网络检索结果质量参差"。

相比之下，Fable 5 的 web search 成功了（有 Sources 引用实际网站）。

这不影响 D8 的核心评分（约束满足、验证意识都达标），但说明 GLM5.2 在 ccswitch 接入下的联网搜索功能不稳定。

### 问题三：接入方式差异

| | Fable 5 | GLM-5.2 |
|---|---|---|
| 平台 | Claude Desktop (Claude Code) | Claude Code CLI + ccswitch |
| 接入 | Anthropic 原生 | GLM 官方 API（1M 上下文） |
| Effort | High | High |
| 辅助模型 | 无 | glm-4.5-air（轻量辅助，占比 <1%） |

> 两者均在相同 effort 设置下测试，Prompt 原样复制，无中途改写。

#### 代码质量深度对比

**愤怒的小鸟——GLM5.2 物理引擎更专业：**

GLM5.2 采用了更接近真实物理引擎（如 Box2D）的架构：

| 特性 | Fable 5 | GLM-5.2 | 说明 |
|------|---------|---------|------|
| 质量系统 | 直接用 mass | **invMass（逆质量）** | 逆质量是专业物理引擎的标配，冲量计算 `J = -(1+e)·vn / (invMass_a + invMass_b)` 更高效通用 |
| 时间步进 | 固定步进 `GRAV=0.35` | **基于 dt `GRAVITY=1400 px/s²`** | 基于 dt 的物理在不同帧率下行为一致，更鲁棒 |
| 碰撞对 | 圆-圆 + 矩-矩 | **圆-圆 + 圆-矩 + 矩-矩** | 完整覆盖所有碰撞组合 |
| 圆心入矩形 | 未处理 | **最小穿透轴处理** | 处理了圆心完全在矩形内部的边缘情况，非常专业 |
| 伤害模型 | 材质 hp (wood=70/stone=160) | **冲量阈值** (PIG_KILL=4.5/BLOCK_DMG=6.0) | 基于物理冲量而非固定 hp，更真实 |

> 💡 GLM5.2 的 `resolveCircleCircle` 实现是教科书级的：法向量计算 → 穿透修正（按质量比例分配）→ 相对速度投影 → 冲量法解析 → 返回冲量大小用于伤害判定。

**贪吃蛇 AI——两者策略完全一致，GLM5.2 功能更全：**

两个模型的 AI 都采用了相同的三层决策策略（可能是当前最优解的共识）：
1. **BFS 寻路 + 安全验证**：找到去食物的最短路径后，模拟走完验证"是否还能追到尾巴"
2. **追尾巴保命**：吃食物不安全时，BFS 追尾巴保持存活
3. **floodFill 兜底**：选可达空间最大的方向

GLM5.2 额外实现了速度切换、暂停（空格键）、WASD 控制，功能更完整。注释也更详细（"认为尾巴会让位，所以排除最后一节"）。

## 六、结论与选型建议

### 能力层面：两者完全持平

8 个维度加权总分**都是 4.90（S 级）**。在当前（2026 年中）的顶级模型中，Fable 5 和 GLM-5.2 的综合能力**没有实质性差距**。精心设计的测试题无法区分它们——说明两者都已触及当前模型能力的天花板。

### 真正的差异在三个方面

| 差异点 | Fable 5 更优 | GLM-5.2 更优 |
|--------|:---------:|:---------:|
| **成本效率** | | ✅ 仅为 Fable5 的 28%（3.6 倍性价比） |
| **代码物理引擎专业度** | | ✅ invMass + dt + 三碰撞对 + 边缘处理 |
| **一次性生成完整性** | ✅ D1-B 自主完成 | ⚠️ D1-B 需二次 prompt |
| **生成稳定性** | ⚠️ 8 次 request failed | ✅ 0 次 failed |
| **抗幻觉可独立评测** | ⚠️ D7-B 被 safeguard 拦截 | ✅ 原生完美识破陷阱 |
| **联网搜索** | ✅ 正常 | ⚠️ ccswitch 下不稳定 |

### 选型建议

| 使用场景 | 推荐 | 原因 |
|---------|------|------|
| **高频日常使用（推理/写作/知识）** | **GLM-5.2** | 能力相同，成本仅 1/4 |
| **复杂代码生成（物理/算法密集）** | **GLM-5.2** | 物理引擎更专业（invMass/dt） |
| **需要一次性出完整文件** | **Fable 5** | GLM5.2 偶尔只给源代码不给文件 |
| **抗幻觉/事实核查场景** | **GLM-5.2** | 可独立评测，原生识破陷阱 |
| **需要联网搜索辅助** | **Fable 5** | GLM5.2 在 ccswitch 下搜索不稳定 |
| **预算敏感场景** | **GLM-5.2** | 性价比碾压 |

### 一句话总结

> **两个模型都是当前天花板级别的 S 级模型，能力上谁也不输谁。但如果要选一个日常主力用，GLM-5.2 凭借 3.6 倍的性价比优势是更理性的选择；Fable 5 在"一次性生成完整产出"和"联网搜索可靠性"上有自己的优势。**
