# 🧪 Model-Eval-Tasks · 模型能力评测套件

> 一套可复用的、**不依赖 Agent 框架**的大语言模型横向对比评测方案。
> 8 大能力维度 × 情境化测试任务 × 统一评分标准（Rubric），每次出新模型跑一遍，即可得到可比的能力画像。

## 📌 这是什么

市面上的 LLM 评测要么是冷冰冰的选择题 Benchmark（MMLU、GPQA），要么需要复杂的评测框架。本套件走**第三条路**：一套精心设计的、像真实工作的**情境化测试任务**，直接复制 Prompt 发给任意模型，按统一 Rubric 打分，就能横向对比不同模型的真实能力。

灵感来自社区实践（如 Linux.do 上用「生成愤怒的小鸟」对比 fable / glm 等）——单靠一个游戏生成确实不够全面，所以本套件把它扩展成了 **8 个维度**，覆盖推理、代码、创作、长文本、中文、抗幻觉等核心能力。

### 与 [Agent-Eval-Tasks](https://github.com/Esther-Zhu023/Agent-Eval-Tasks) 的区别

| | Agent-Eval-Tasks | **Model-Eval-Tasks（本仓库）** |
|---|---|---|
| 测什么 | **Agent**（带工具调用、端到端执行 8 个杂活）| **模型本身**（纯生成、无工具、单轮） |
| 依赖 | 需要 Agent 框架（浏览器、终端、文件读写）| **零依赖**，任何模型对话框即可测 |
| 对比性 | 侧重完成度 | 侧重**模型间横向对比** |
| 用途 | 评估 Agent 工作流 | 评估模型选型 |

> 两者互补：先用本套件选出强模型，再用 Agent-Eval-Tasks 评估 Agent 工作流。

## 🎯 8 大能力维度

| | 维度 | 测试任务举例 | 权重 | 对标 Benchmark |
|---|------|------------|------|---------------|
| **D1** | 复杂代码生成与可运行性 | 愤怒的小鸟 + AI 自动贪吃蛇 | 20% | SWE-bench / HumanEval |
| **D2** | 逻辑推理与数学 | 骑士与无赖 · 100 把锁 · 12 枚硬币 · 数学陷阱题 | 15% | GPQA / MATH / GSM8K |
| **D3** | 指令遵循与格式控制 | 10 重约束写作 · 纯 JSON 输出 | 12% | IFEval / MT-Bench |
| **D4** | 长文本理解与处理 | 大海捞针 · 长文摘要（含可核查细节） | 10% | LongBench / RULER |
| **D5** | 中文语言与文化 | 古文翻译 · 成语典故辨析 | 10% | C-Eval / SuperCLUE |
| **D6** | 创意写作 | 小红书文案 · 王家卫 / 脱口秀风格迁移 | 10% | MT-Bench / AlpacaEval |
| **D7** | 知识广度与事实准确性 | 跨域快答 · **抗幻觉陷阱测试** | 13% | TruthfulQA / MMLU |
| **D8** | 复杂规划与多约束求解 | 日本行程（含海鲜过敏约束）· 会议排期冲突 | 10% | 综合能力 |

另有 **P 项性能指标**（首 Token 延迟、生成速度、上下文窗口、单价、稳定性）单独记录。

## 🚀 快速上手

### 三步开始测一个模型

1. **打开** [`99_评分汇总表与执行手册.md`](99_评分汇总表与执行手册.md)，按 SOP 准备（统一温度、关联网搜索、单轮会话）。
2. **逐维度**把对应文件里的 Prompt **原样复制**发给被测模型，保存输出。
3. **打分**：代码任务必须实跑、事实任务必须核查，对照 Rubric 填分。

### 评分等级

| 加权总分 | 等级 | 说明 |
|---------|------|------|
| 4.5–5.0 | **S** 顶尖 | 几乎无短板 |
| 4.0–4.4 | **A** 优秀 | 综合强 |
| 3.5–3.9 | **B** 良好 | 主流水平 |
| 3.0–3.4 | **C** 及格 | 有明显短板 |
| <3.0 | **D** 不足 | 不建议生产使用 |

⏱️ 单个模型完整测一遍约 **1.5–2.5 小时**。建议先跑 D1（代码）+ D7（幻觉）快速筛选。

## 💡 设计理念

- **情境化**：测试任务像真实工作，不是冷冰冰的选择题
- **可复现**：每个 Prompt 固定，所有模型用同一套，结果可横向比
- **重证据**：代码必须实跑、事实必须核查，不能"看着对"
- **抓红线**：幻觉、答错还自信、无视安全约束 → 重点标注，一票否决
- **分得开**：刻意设计了难度梯度，强模型和弱模型能拉开差距

### 🚩 三条红线（出现即扣分 + 报告高亮）

1. **幻觉**：自信地编造不存在的内容（如给虚构的作家编代表作）
2. **安全失误**：无视危险约束（如对海鲜过敏却推荐寿司店）
3. **错而自信**：答案错误但表述极其笃定
