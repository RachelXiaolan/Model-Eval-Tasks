# 🧪 模型能力评测套件 v1.0

> **用途**：一套可复用的、不依赖 Agent 框架的**模型本身能力**横向对比方案。
> 每次出新模型，跑一遍这套任务，按统一 Rubric 打分，即可得到可比的能力画像。
>
> **设计参考**：
> - Esther 的 [Agent-Eval-Tasks](https://github.com/Esther-Zhu023/Agent-Eval-Tasks)（情境化任务的形式）
> - 主流 Benchmark：MMLU / GPQA / MATH / HumanEval / SWE-bench / IFEval / MT-Bench / C-Eval / LongBench / TruthfulQA / Chatbot Arena
> - 社区实战：Linux.do「愤怒的小鸟生成」「十三模型横向对比」等
> - 学术综述：《A Survey on Evaluation of Large Language Models》(微软亚研, 219 篇文献)

---

## 一、怎么用这套方案

### 执行流程（SOP）

1. **统一条件**：所有被测模型使用相同的系统提示词（或不用）、相同的温度（建议 `temperature=0.7` 用于创作类、`0` 用于事实/推理类）、相同的会话设置（单轮、无联网、无工具调用）。
2. **逐题跑**：把本文件的每个测试 Prompt **原样**发给被测模型，一次一个，记录完整原始输出。
3. **留存证据**：代码类任务必须**实际运行**验证；生成类任务保留原文。每项截图/存档。
4. **打分**：对照该维度的 Rubric，逐项给 0–5 分，填入评分表。
5. **出报告**：汇总成能力雷达图 + 总分排名。

### 关键原则

- ⚠️ **必须实跑代码**：生成出来的游戏/程序，到底能不能跑、好不好玩，必须实际打开验证，不能只看代码长得对不对。
- ⚠️ **事实题必须核查**：知识类问题的答案，必须自己 Google/查证，不能模型说什么信什么。
- ⚠️ **同一题给所有模型完全相同的 Prompt**，不要中途改写。把 Prompt 直接复制粘贴。
- ⚠️ **盲评**：打分时建议先遮住模型名字，避免品牌偏见（尤其测自家/友商模型时）。

## 二、评测维度总览

| 维度 | 考什么 | 对标 Benchmark | 权重 |
|------|--------|---------------|------|
| **D1** 复杂代码生成与可运行性 | 一次性生成完整可运行的程序（游戏/工具） | SWE-bench / HumanEval / 社区实战 | 20% |
| **D2** 逻辑推理与数学 | 演绎推理、多步逻辑、数学求解 | GPQA / MATH / GSM8K / BBH | 15% |
| **D3** 指令遵循与格式控制 | 多重约束、精确格式、负面指令 | IFEval / MT-Bench | 12% |
| **D4** 长文本理解与处理 | 长文摘要、大海捞针、跨段推理 | LongBench / RULER / LongMemEval | 10% |
| **D5** 中文语言与文化 | 中文创作、成语、古文、文化理解 | C-Eval / SuperCLUE / CMMLU | 10% |
| **D6** 创意写作 | 故事、文案、风格迁移、情感表达 | MT-Bench / AlpacaEval | 10% |
| **D7** 知识广度与事实准确性 | 跨域知识、事实核查、抗幻觉 | MMLU / TruthfulQA / HLE | 13% |
| **D8** 复杂规划与多约束求解 | 行程规划、资源分配、冲突协调 | — (综合能力) | 10% |

**附加 P 项 — 性能与非能力指标**（不计入能力分，单独列）

| 指标 | 说明 |
|------|------|
| P1 首 Token 延迟 (TTFT) | 从发出到第一个字出现的时间 |
| P2 生成速度 (tokens/s) | 吞吐量 |
| P3 有效上下文窗口 | 实际能正确处理的最大长度 |
| P4 单价 | 每百万 token 成本 |
| P5 稳定性 | 长任务中断/超时/限频概率 |

> **总分计算**：Σ(各维度得分 × 权重) = 加权总分（0–5 分制）。性能指标单独展示，不进总分。
